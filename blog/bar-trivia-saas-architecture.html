Architecture of a real-time AI trivia SaaS: Barplay teardown

Barplay lets bar venues run AI-powered trivia nights. Players scan a QR code, join from their phone, and compete on Claude-generated questions in real time. Full stack breakdown.

Most bar trivia is a pain to run. The host has to source questions, manage paper score sheets, and deal with disputes. We built Barplay to eliminate all of that: a venue gets a QR code, players join from their phones, and the entire game — question generation, scoring, leaderboard — runs automatically.

This is a technical breakdown of how it's built. The code is complete; we're waiting on deployment credentials to take it live.

The product in 30 seconds

  • Venue signs up, gets a dashboard, configures trivia settings (category, difficulty, number of questions)
  • Starts a session → QR code appears on the bar's screen
  • Players scan, enter a name, play from their phone
  • Questions stream from Claude in real time with a 30-second timer
  • Live leaderboard updates after each question
  • Venue pays €29–79/month depending on session volume

Stack overview

API

Hono on Node.js — lightweight, fast, TypeScript-first. Handles REST routes for sessions, answers, questions, venues, and Stripe webhooks.

Frontend

Next.js 14 App Router. Player flow is a single route with state machine: name entry → question → result → leaderboard. SSE for real-time pushes.

Database

Supabase (PostgreSQL + pgvector). Supabase Realtime handles leaderboard broadcasting without a separate WebSocket server.

AI

Claude Sonnet for generation, Claude Haiku for validation. Two-call pattern per question. pgvector cosine similarity for deduplication.

Queue

BullMQ + Upstash Redis. Question generation jobs pre-warm 10 questions per session asynchronously before the game starts.

Billing

Stripe Subscriptions + webhooks. Tier enforcement at the API layer based on current subscription status.

Auth

Clerk for venue accounts. Players are anonymous (name + session ID stored in localStorage).

Deploy

Railway — two services (API, web), automatic deploys from GitHub main branch.

The AI question pipeline

This is the technically interesting part. Generating questions is easy. Generating good, non-duplicate questions at scale is harder.

Two-model pattern

We use Claude Sonnet to generate and Claude Haiku to validate. The generation prompt asks for question + 4 answer choices + correct answer + explanation. The validation prompt checks:

  • Is the correct answer actually correct?
  • Are the distractors plausible but clearly wrong?
  • Is the question unambiguous?

Haiku rejects ~15% of Sonnet's output. The questions that pass are noticeably better than single-model generation.

pgvector deduplication

We embed each approved question using a text embedding model and store the vector in Supabase. Before any new question is used, we query for cosine similarity against all questions in the venue's history:

SELECT id, question_text, 1 - (embedding <=> $1) AS similarity FROM questions WHERE venue_id = $2 AND 1 - (embedding <=> $1) > 0.85 LIMIT 1;

If similarity > 0.85 with any existing question, we regenerate. This means even if Claude produces a semantically identical question with different wording ("What year was X founded?" vs "In which year was X established?"), it gets caught.

Pre-warming with BullMQ

We don't generate questions on-demand during a game — that would add 2-3 seconds of latency per question. Instead, a BullMQ job fires when a session starts and pre-generates 10 questions asynchronously. The game can start after the first 3 are ready; the rest arrive before they're needed.

Real-time architecture

Supabase Realtime is doing the heavy lifting. The leaderboard table in PostgreSQL has a Realtime subscription; any INSERT or UPDATE to player scores broadcasts to all connected clients in the session. We get WebSocket-like behavior without running a WebSocket server.

The question timer runs client-side (no server round-trips), synced at session start. Answers POST to the API within the 30-second window; late answers are rejected at the route level.

Billing and tier enforcement

Three tiers:

  • Starter (€29/mo): 4 sessions/month, 20 players/session
  • Pro (€49/mo): 15 sessions/month, 50 players/session
  • Unlimited (€79/mo): No limits

Stripe webhooks update a subscriptions table in Supabase on every payment event. The API checks tier limits before allowing session creation. No client-side gating — the enforcement is entirely server-side.

Why this stack for an MVP? Hono + Next.js + Supabase eliminates the need for a separate WebSocket server, a separate auth service, and a separate file storage service. Supabase does database, auth (via Clerk integration), storage, and realtime. Railway handles deploy in one click. The whole infrastructure fits on two Railway services and one Supabase project.
