We built an AI content machine that runs without human input. Here's the stack.

Daily X threads, LinkedIn posts, and blog articles — generated by Claude, published automatically. Zero editors. Zero briefing sessions.

The brief was simple: Factoria needs content shipping every day to build an audience for our SaaS products. But hiring writers is slow and expensive. Social media managers are slower. So we automated the whole thing.

Here's what runs every day at 7am UTC without anyone touching a keyboard.

The architecture

The pipeline is a TypeScript Node.js service with four components:

  1. Generator — calls Claude Haiku with a structured prompt, returns typed JSON
  2. Validators — Zod schemas enforce character limits and structure per platform
  3. Publishers — platform-specific adapters (X OAuth 1.0a, Ghost Admin API)
  4. Scheduler — Claude Code remote trigger on cron, zero infrastructure

The entire thing is ~400 lines across 4 files. No database. No queue. No Docker. Just a TypeScript process that runs, generates, publishes, and exits.

The generation prompt

Most AI content pipelines fail because they generate the same copy for every platform and call it "distribution." That's not how content works. A LinkedIn post and a tweet thread are completely different formats with different native grammars.

We give Claude a single core angle and ask it to produce platform-native output for each:

{ "angle": "One sentence: the core idea", "x_standalone": "Punchy single tweet ≤280 chars", "x_thread": { "hook": "Scroll-stopping opener", "tweets": ["value", "value", "value"], "cta_tweet": "Small clear ask" }, "linkedin": { "opening_line": "Preview text — must work alone", "body": "Short paragraphs, lesson framing", "cta": "One clear ask" }, "blog": { "title": "SEO title", "body": "700-1000 word markdown", "tags": ["tag1"], "meta_description": "≤160 chars" } }

The constraint "same idea, different format" is the key instruction. Without it, you get four slightly different versions of the same paragraph.

Why Claude Haiku, not Sonnet

This runs every day. At Sonnet pricing, that's ~$0.50/day for a task that doesn't require deep reasoning — just good writing with clear structure. Haiku costs roughly 10x less and produces output that's indistinguishable for this use case.

We reserve Sonnet (and Opus) for tasks that actually need them: code generation, complex analysis, multi-step agent reasoning. Content generation with a clear format spec isn't one of them.

The scheduling layer

The pipeline runs as a Claude Code Remote Trigger — essentially a fully isolated Claude session that spawns on a cron schedule, executes, and terminates. No server to maintain. No uptime to worry about. Cost is zero when it's not running.

This is the right architecture for any scheduled AI task that doesn't need persistent state: daily reports, weekly digests, content generation, data syncs.

What it produces

Every day, without fail:

  • 1 standalone X tweet on a specific, non-generic angle
  • 1 X thread (6-8 tweets) with real value per tweet
  • 1 LinkedIn post with native formatting (short paragraphs, explicit lesson framing)
  • 1 blog post (700-1000 words, SEO title, meta description)

The content covers the "AI tools and automation for entrepreneurs" niche because that's our target audience. Every piece of content ends with a CTA back to Factoria.

What's not automated yet

Publishing is still semi-manual — we haven't wired the X OAuth credentials or Ghost API key to the live scheduler yet. Right now content drops into run logs; manual push to social. It's on the roadmap.

The honest take: even without auto-publishing, having the daily generation running means content exists. The bottleneck shifted from "creating" to "distributing" — which is a much faster problem to solve.
